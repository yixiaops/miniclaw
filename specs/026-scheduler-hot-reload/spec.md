# Feature Specification: 定时任务与动态配置加载

**Feature Branch**: `026-scheduler-hot-reload`  
**Created**: 2026-05-15  
**Status**: Draft  
**Input**: User description: "为 Miniclaw 添加两个功能：1) 定时任务功能 - 支持通过自然语言设置定时任务，让 AI 在指定时间执行特定任务；2) 动态配置加载 - 支持实时加载新增的子 agent 配置，无需重启服务"

## Clarifications

### Session 2026-05-15

- Q: 用户离线时的任务处理方式 → A: 消息持久化，用户上线后自动推送
- Q: 重复任务的去重策略 → A: 智能合并：相同时间+相似内容时合并，AI 询问确认
- Q: 定时任务执行内容类型 → A: 提醒消息 + 可执行预设指令（如生成周报、调用子 Agent）
- Q: 任务访问权限隔离 → A: 完全隔离：用户只能查看/管理自己创建的任务
- Q: 周期性任务的终止条件 → A: 默认无限制执行，用户手动取消终止

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 自然语言创建定时提醒 (Priority: P1)

用户在日常对话中，通过自然语言告诉 AI："明天早上9点提醒我开会"。AI 理解意图，解析时间表达式，创建定时任务。当时间到达时，AI 主动向用户发送提醒消息。

**Why this priority**: 这是用户最常用的场景，体现了"自然语言交互"的核心价值，可以独立交付并验证。

**Independent Test**: 用户发送"明天早上9点提醒我开会"，系统在指定时间向用户发送提醒消息。可通过手动触发定时器验证。

**Acceptance Scenarios**:

1. **Given** 用户正在与 AI 对话，**When** 用户说"明天早上9点提醒我开会"，**Then** AI 确认任务创建成功，并在指定时间向用户发送提醒
2. **Given** 用户已创建定时任务，**When** 用户说"取消明天的提醒"，**Then** AI 删除对应任务并确认
3. **Given** 用户说"每隔周一上午10点生成周报"，**When** AI 解析成功，**Then** 创建周期性任务并在每个周一执行

---

### User Story 2 - 查看和管理定时任务列表 (Priority: P2)

用户可以随时查看当前所有定时任务，了解任务内容、执行时间和状态。

**Why this priority**: 管理功能是必要补充，但依赖 P1 的任务创建功能。

**Independent Test**: 用户发送"查看我的定时任务"，AI 返回任务列表（包含任务内容、下次执行时间）。

**Acceptance Scenarios**:

1. **Given** 用户有多个定时任务，**When** 用户说"看看我的任务列表"，**Then** AI 返回所有任务的摘要信息
2. **Given** 用户没有定时任务，**When** 用户请求查看任务列表，**Then** AI 返回"您目前没有定时任务"

---

### User Story 3 - 动态添加子 Agent (Priority: P1)

运维人员在生产环境中添加新的子 Agent 配置文件（如 `etf-analyst.md`），系统自动检测并加载新 Agent，无需重启服务。用户在下一次对话中即可使用新 Agent。

**Why this priority**: 与 P1-定时任务同等重要，可独立开发和验证，减少运维负担。

**Independent Test**: 在 `~/.miniclaw/prompts/` 目录添加新的 Agent 配置文件，用户发送消息触发子 Agent 调用时，系统成功加载并执行新 Agent。

**Acceptance Scenarios**:

1. **Given** 服务正在运行，**When** 运维人员添加新 Agent 配置文件到配置目录，**Then** 系统在5秒内检测到变更并加载新 Agent
2. **Given** 新 Agent 已加载，**When** 主 Agent 收到匹配该 Agent 能力的用户请求，**Then** 主 Agent 可以成功调用新创建的子 Agent
3. **Given** 配置文件格式有误，**When** 系统尝试加载，**Then** 记录错误日志但不影响现有 Agent 运行

---

### User Story 4 - 动态修改 Agent 配置 (Priority: P2)

运维人员修改现有 Agent 的系统提示词或工具配置，系统自动重新加载，后续对话立即生效。

**Why this priority**: 增强运维灵活性，但依赖 P3 的配置检测机制。

**Independent Test**: 修改现有 Agent 的配置文件，用户下次对话中观察到新配置生效。

**Acceptance Scenarios**:

1. **Given** Agent A 已加载，**When** 运维人员修改 Agent A 的系统提示词，**Then** Agent A 在下次对话中使用新提示词
2. **Given** Agent 的工具列表被修改，**When** 配置重新加载，**Then** Agent 的可用工具列表更新

---

### Edge Cases

- 用户说的自然语言时间表达式无法解析时（如"下个蓝月亮提醒我"），AI 请求用户澄清或拒绝创建
- 定时任务执行时，用户 Session 不活跃（如飞书渠道用户离线），消息持久化存储，等待用户上线后自动推送
- 配置文件被删除时，正在执行的 Agent 继续完成当前任务，新 Session 无法创建该 Agent
- 配置加载失败时，记录错误日志并可选发送告警通知（需配置告警渠道）
- 多个定时任务同时触发时，按任务创建顺序依次执行，避免并发冲突

## Requirements *(mandatory)*

### Functional Requirements

**定时任务模块:**

- **FR-001**: 系统必须能够从用户自然语言中识别定时任务意图（关键词：提醒、定时、每天、每周、几点等）
- **FR-002**: 系统必须支持解析常见时间表达式：相对时间（明天、下周）、绝对时间（5月20日）、周期时间（每天、每周）
- **FR-003**: 系统必须支持解析具体时刻：上午/下午 + 具体时间（如"早上9点"、"下午3点半")
- **FR-004**: 系统必须将定时任务持久化存储，重启服务后任务不丢失
- **FR-005**: 系统必须支持定时任务的创建、删除、修改、查询操作
- **FR-006**: 系统必须支持一次性任务和周期性任务两种类型
- **FR-006a**: 系统必须支持智能去重：当新任务与现有任务时间相同（误差≤30分钟）且内容相似时，AI 询问用户是否合并而非重复创建
- **FR-006b**: 系统必须支持两种任务动作类型：提醒消息（纯文本推送）和预设指令执行（如调用子 Agent、生成报告）
- **FR-006c**: 系统必须实现任务访问权限完全隔离：用户只能查看、修改、删除自己创建的任务，无法访问其他用户的任务
- **FR-006d**: 周期性任务默认无限制执行，直到用户明确取消；用户可通过自然语言取消周期性任务
- **FR-007**: 定时任务执行时，系统必须向原用户发送消息（通过原渠道）；若用户离线，消息必须持久化存储，等待用户上线后推送
- **FR-008**: 系统必须支持任务执行失败时的重试机制（最多3次）
- **FR-009**: 系统必须提供任务状态追踪：待执行、已执行、已取消、执行失败、等待推送（用户离线）

**动态配置加载模块:**

- **FR-010**: 系统必须监听 Agent 配置目录（`~/.miniclaw/prompts/` 和 `~/.miniclaw/config.json`）
- **FR-011**: 系统必须在检测到配置文件变更后5秒内完成重新加载
- **FR-012**: 系统必须支持加载新增的 Agent 配置
- **FR-013**: 系统必须支持重新加载已修改的 Agent 配置
- **FR-014**: 系统必须在配置加载失败时记录错误日志，不影响现有 Agent
- **FR-015**: 系统必须支持配置文件删除时清理对应 Agent 实例
- **FR-016**: 配置重新加载后，新 Session 必须使用新配置；现有活跃 Session 可继续使用旧配置直到 Session 结束

### Key Entities

- **ScheduledTask**: 定时任务实体，包含任务ID、用户标识（用于权限隔离，用户仅能访问自己的任务）、渠道、任务内容、内容摘要（用于去重匹配）、执行时间（cron表达式或具体时间）、任务类型（一次性/周期性）、动作类型（提醒/指令）、动作参数（可选，如目标Agent ID）、状态、创建时间、重试次数
- **TaskExecutionLog**: 任务执行日志，记录每次执行的时间、结果、错误信息
- **AgentConfig**: Agent 配置实体，包含配置ID、配置路径、最后加载时间、加载状态、配置内容摘要
- **ConfigChangeEvent**: 配置变更事件，记录变更类型（新增/修改/删除）、变更时间、涉及的配置文件

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户通过自然语言创建定时任务的成功率达到 90% 以上（成功定义：AI 正确解析时间并确认创建）
- **SC-002**: 定时任务在预定时间的 1 分钟内准时触发执行
- **SC-003**: 新增 Agent 配置文件后，系统在 5 秒内完成加载，用户可立即调用新 Agent
- **SC-004**: 配置重新加载过程中，现有对话不受影响，服务无中断
- **SC-005**: 定时任务支持至少 100 个并发任务，任务执行成功率 95% 以上
- **SC-006**: 配置加载错误时，系统在日志中记录详细错误信息，现有 Agent 继续正常运行

## Assumptions

- 时间表达式解析依赖 LLM 的自然语言理解能力，不使用外部 NLP 库
- 定时任务使用 Node.js 内置的定时机制或 node-cron 库实现
- 配置监听使用 Node.js 的 fs.watch 或 fs.watchFile API
- 飞书渠道的离线消息通过飞书 API 的推送能力实现（用户在线时收到，离线时等待上线）
- Agent 配置文件格式延续现有 YAML frontmatter 格式
- 定时任务的持久化存储使用 JSON 文件，路径为 `~/.miniclaw/scheduled-tasks.json`
